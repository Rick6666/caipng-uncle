## 3. 菜品表（dishes）

备菜阶段将食材烹为菜品，1 份菜消耗配方内每种食材各 1 单位。

> **日常菜（3 蔬 + 3 蛋 + 4 荤 = 10 道）开局全部可做**；只有 2 道海鲜按声望解锁

### 4.2 玄学报价机制（核心）

订单基准价 `base = RICE_PRICE + Σ(菜品基准价)`。三档报价：

| 档位 | 价格公式 | 说明 |
|---|---|---|
| kind 良心价 | max(cost+1, round(base × 0.8)) | 少赚，赚口碑 |
| normal 正常价 | base | 标准 |
| slash 斩客价 | round(base × 1.5) | 多赚，看人下菜 |

**反应矩阵**（按顾客类型 × 档位；`pay` = 付款概率，付款失败 = 掉头走且当单菜品已出、计损耗）：

> **设计原则（v1.1 修订，见 grilling 会话结论）**：每一档对每一类顾客都必须有存在的理由，
> 不允许出现"正常价在同等付款概率下 rep 和金钱双项都不劣于良心价"这种无脑正解。
> 三档分层为三种打法：**kind = guaranteed 高声望**、**normal = guaranteed 均衡**、
> **slash = 高风险**——其中 slash 对"好斩"型顾客（worker/labourer）期望金钱 > normal，
> 对"精明"型顾客（student/ahma/foodie）期望金钱 < normal，玩家需要**认人下菜**而非套用统一策略。

| 类型 | kind（0.8×base） | normal（1.0×base） | slash（1.5×base） | 期望值备注 |
|---|---|---|---|---|
| student | pay 1.0, rep +2 | pay 0.9, rep 0 | pay 0.3, 付款则 rep −3 / 走人 rep −2 | slash 期望金钱 0.45×base ＜ normal 0.9×base，别斩 |
| worker | pay 1.0, rep +1 | pay 1.0, rep 0 | pay 0.85, 付款则 rep −1 / 走人 rep −2 | slash 期望金钱 1.275×base ＞ normal 1.0×base，好斩 |
| ahma | pay 1.0, rep +3 | pay 0.9, rep 0 | **触发砍价子选择**（见下） | — |
| uncle2 | pay 1.0, rep +2 | pay 1.0, rep 0 | pay 0.5, rep −2 / 走人 rep −2 | 纯 money(1.0×) vs rep(+2) 二选一，无赌博项，新手教学 |
| labourer | pay 1.0, rep +2 | pay 1.0, rep +1 | pay 0.75, 付款则 rep −1 / 走人 rep −2 | slash 期望金钱 1.125×base ＞ normal 1.0×base，好斩 |
| influencer | pay 1.0, **rep +6**（发好评帖） | pay 1.0, rep +1 | pay 1.0, **rep −6**（发避雷帖） | 付款恒定 1.0，三档纯粹是声望赌注大小，非金钱博弈 |
| foodie | pay 1.0, rep +4 | pay 1.0, rep +2 | pay 0.6, rep −5 / 走人 rep −4 | slash 期望金钱 0.9×base ＜ normal 1.0×base，懂行别斩 |

关键修正对比旧版：worker 的 normal 档去掉了"小费+$1"且 rep 改为 0（原设计 kind/normal 同为 rep+1 但 normal
金钱更高，导致 kind 无存在意义）；worker/labourer 的 slash 付款概率从 0.8/0.7 上调至 0.85/0.75，
使其期望金钱反超 normal，二者成为游戏内明确的"好斩"目标，与 student/ahma/foodie 的"别斩"型形成
认知反差——这正是"看人下菜"梗的策略落点：玩家要记住谁好糊弄、谁不好惹，而不是无脑同一档位打天下。

**阿嬷砍价子选择**（选斩客价时弹出）：
- 「好啦好啦算你正常价」→ 收 normal 价，rep 0；
- 「阿嬷这个真的不能少」→ 0.5 概率照付（rep −1）/ 0.5 概率走人（rep −3）。

此子选择两个分支期望值均略低于直接接受 normal 价——刻意保留为"贪心赌一把"的负 EV 选项，
不是平衡缺陷：面向愿意为博弈爽感牺牲期望值的玩家，且触发频率低（仅 ahma 一种顾客的 slash 分支），
不构成主导策略。

**美食家追加判定**：付款后，若当日备菜种类 ≥ 8 → 额外 rep +4（"菜色齐全，正宗！"）。

### 4.3 缺菜处理

顾客点的菜若已售罄（点单时按备菜抽取，营业中可能被前面顾客买光——点单快照在顾客生成时，
出餐时校验），玩家二选一：
- **推荐替代**：从同类别现存菜品中推一道，0.6 概率接受（订单替换后进报价流程）；拒绝则走人 rep −1；无同类菜时此选项置灰。
- **道歉送客**：rep −1，顾客离开。

### 4.4 每日客流公式

```
n = BASE_CUSTOMERS + floor(rep / 8) + (霓虹招牌 ? 2 : 0) + randInt(-1, +2)
雨天且无遮阳棚: n = floor(n × 0.7)
n = clamp(n, 3, MAX_CUSTOMERS + (帮手 ? 4 : 0))
```

## 8. 经济结算（收档顺序，必须严格按此顺序实现）

1. 收档事件判定（inspection / catSteal）；
2. 剩熟菜处理：无冰箱 → 全部作废；有冰箱 → 每种保留 `floor(qty × 0.6)`；
3. 扣房租 RENT_PER_DAY；
4. 若有未还贷款：扣 LOAN_INTEREST 计入已还额，还满 LOAN_REPAY 自动结清；
5. 破产判定：`money < 0`：
   - 从未贷款 → 触发阿财叔剧情，money += LOAN_AMOUNT，进入还款状态；
   - 已有贷款或已用过 → **游戏结束（破产结局，展示 §9.2 墓志铭，不是 §9 评分页）**；
6. 累计 `state.stats`（本局唯一的统计，供 §9 结局评分与 §9.1 人设标签使用）；
7. 展示当日报表（营收/成本/损耗/房租/利息/声望变化/高光台词）；进入 `settle` 阶段等待 `ACK_SETTLE`
   （由 `03-architecture.md` §4 决定：day < GAME_DAYS → phase='shop'；day = GAME_DAYS → phase='ending'）。

## 9. 结局评分（第 7 天收档后，一局唯一的终点）

> **v1.3 修订（grilling 会话结论）**：此前讨论过"30 天赛季"与"7 天周报+连续经营"两个方向，
> 均已废弃。最终方向更彻底：**一局就是固定的 7 天，讲完即结束，没有"继续经营""下一周""赛季"
> 这些概念**——对标 Reigns/Suika Game 这类单局可坐着通关、局与局之间互不延续的成瘾型休闲游戏。
> money / rep / upgrades 只在本局 7 天内有意义，重开新局即完全归零，**不支持中途保存/恢复**，
> 这是刻意的简化取舍：关闭页面 = 本局作废，只能「重新开始」。

**结局评分**（评的是这一整局 7 天的经营表现）：

```
score = money + rep × 5
```

| 评分 | 阈值（经声望经济重平衡后 `sim.js` 校准：survivor 分数分布 ~79~283，中位 ~157） |
|---|---|
| S | ≥ 220 |
| A | ≥ 150 |
| B | ≥ 100 |
| C | < 100 |

结局页展示：最终评分、经营统计摘要（总接客数/总营收/最赚一天/被骂次数等，取自 `state.stats`）、
§9.1 人设标签，以及一句本局高光/翻车事件引用。之后**只有一个按钮**：「再来一局」（`NEW_GAME`，
完全重新开始，不带任何本局数值）。

`rep ≥ 120` 不弹窗、不打断当日营业——就是结局评分之外，结局页里附加展示的一档特殊评价
（"这局你干到了全城最强杂菜饭"），和 S 档评分是同一时刻展示的两件事，不是单独的流程分支。

### 9.1 Uncle 人设标签（传播钩子）

结局页在最终评分之外，额外生成一个基于本局打法风格的"人设标签"，类似年度报告式的传播设计——
玩家倾向截图分享"我这局是哪种 Uncle"这个身份认同结果，而不是单纯的分数。

**计算依据**：`state.stats`（已在 `03-architecture.md` §3 定义，本局唯一的统计，无需新增字段）：

```
slashRate = stats.slashCount / max(1, stats.totalServed)
walkoutRate = stats.walkoutCount / max(1, stats.totalServed)
```

**判定优先级（自上而下，命中即止，只取第一条）**：

| 优先级 | 条件 | id | 标签 | flavor |
|---|---|---|---|---|
| 1 | slashRate ≥ 0.4 | `shark` | 🦈 奸商 Uncle | "逢人就宰，回头客都被你吓跑了" |
| 2 | walkoutRate ≥ 0.2 | `awkward` | 😅 社死 Uncle | "被气走的顾客比留下的还多，风评有点危险" |
| 3 | slashRate ≤ 0.05 且 rep ≥ 60 | `kind` | 😇 良心 Uncle | "几乎不斩客，街坊都念你的好" |
| 4 | money < 200 且 rep < 30 | `broke` | 😩 苦命 Uncle | "惨淡经营，还在摸索门道" |
| 5 | 以上均不命中（默认） | `worldly` | 😎 江湖 Uncle | "精明世故，眼里有生意也有人情" |

阈值为初始估计，需在 Task 9（`sim.js`）跑完 `reasonableBot`/`lazyBot`/`slasherBot` 三种基线后校准：
理想情况下 reasonableBot 应落在「江湖」或「良心」，lazyBot 落在「苦命」，slasherBot 落在「奸商」——
若模拟结果显示某个 bot 没有落进预期标签，回来调整本节阈值（同 §11 平衡基线一样，是需要迭代校准的活文档）。

标签**只是展示性 flavor，不影响任何数值**（不解锁内容、不影响下一局），纯粹为传播设计。

### 9.2 阵亡结局：墓志铭系统（v1.5 修订，grilling 会话新增）

> **本局定位从"大概率能过关"改为"失败是常态"**：参考同类型爆火产品《赛博徒步·生死鳌太线》
> 的核心洞察——死亡本身如果好笑、好截图，就是最强的传播素材，"这次能不能撑过去"的悬念
> 比"这次能考多少分"更能驱动"再试一次"。**只抄这套哲学，不抄它的四维生存值/装备子系统**——
> 我们仍然只用 money/rep 两条数值轴，破产判定逻辑（§8）完全不变，只是把"破产"从边缘情况
> 提升为结局设计的第一等公民，专属处理，不能和"正常撑完 7 天"共用一套结局页。

破产结算（§8 步骤 5 命中"已有贷款或已用过"分支）触发时，不显示 §9 的评分/人设标签
（那套是为"活到第 7 天"设计的），改为展示**墓志铭**：根据死于第几天 + 本局打法生成一句
黑色幽默文案，逻辑与 §9.1 uncleTitle 同构（同样是纯展示 flavor，不影响下一局）：

```
slashRate = stats.slashCount / max(1, stats.totalServed)
```

**判定优先级（自上而下，命中即止，只取第一条）**：

| 优先级 | 条件 | id | 称号 | 墓志铭 |
|---|---|---|---|---|
| 1 | day ≤ 2 | `early` | 💀 早夭 Uncle | "接手摊子没两天就撑不住，邻居 Uncle 看了直摇头" |
| 2 | slashRate ≥ 0.3 | `karma` | 🦈 现世报 Uncle | "一路斩客斩上瘾，资金链没扛住，也是活该" |
| 3 | day ≥ 6 | `soClose` | 😭 一步之遥 Uncle | "就差一两天就能全须全尾交摊子了，天不助我" |
| 4 | 以上均不命中（day 3~5 且非高 slash） | `honest` | 😮‍💨 苦撑 Uncle | "老老实实做生意，还是没扛过这个坎" |

结局页展示：`epitaph(day, stats)` 的称号 + 墓志铭、撑到第几天、本局统计摘要，**只有一个按钮**
「再来一局」（同 §9，`NEW_GAME`）。**不做分享按钮**——墓志铭本身就是为截图分享设计的文案，
玩家会自发截图，不需要额外的分享入口（对应 `01-PRD.md` 已从功能清单中移除种子分享 UI）。

### 9.3 每日节奏反馈（不打断流程，只是 flavor）

两处轻量反馈，均不产生新的 phase、不提供选择分支，只是展示性文本：

1. **清晨/备菜阶段客流预热**：展示"预计客流：约 N±2 位"，`N` 直接读 `dailyCustomerCount` 的
   期望值（该函数在 `03-architecture.md` §4 里本来就标注了"供预览"用途，之前没有真正用在 UI
   上）。让玩家在备菜前有个粗略参考，尤其第 1 天完全没有历史数据可依据，这一点尤为重要——
   不改变客流公式本身的随机性（真实客流仍受 randInt(-1,+2) 影响，展示的是期望值不是精确数）。
2. **每日结算报表内的手感评价**：用当天（不是全局）的收支和声望变化做一个三档简单判定，
   写在 `.ledger` 报表下面一行：

   ```
   todayProfit = today.revenue − today.spend
   todayProfit > 0 且 today.repDelta > 0  → "今天生意红火，Uncle 看了都点头"
   todayProfit < 0 或 today.repDelta < -3 → "今天有点惨淡，得想想办法了"
   其余                                    → "今天马马虎虎，混过去了"
   ```

   这不是跨天比较、不是"你和平均水平比怎么样"，只是"今天"这一天内部收支/声望是否健康——
   给玩家一点即时手感反馈，防止在"随时可能破产"的高致命性设定下完全摸黑走到最后一秒。

## 10. 文案风格规范

- 中文为主，混少量 Singlish 词（leh / lah / lor / paiseh / shiok / walao），每句最多 1 个，不堆砌；
- 顾客每类型至少 5 条进场台词、3 条各档位反应台词，随机抽取；
- Uncle 旁白第二人称（"你手起勺落……"）；
- 所有文案集中于 `data.js` 的 `lines` 字段，禁止散落在逻辑代码中。

## 11. 数值平衡基线（供 sim.js 回归，详见 04-test-plan §4）

模拟窗口就是 **GAME_DAYS = 7 天**，与游戏内实际长度完全一致（不再需要"更长的测试 horizon"这个概念，
因为游戏本来就只有 7 天）。这是本项目最关键的平衡校准点——因为每局只有一次、7 天就讲完，
**不存在"前几天不平衡、后面慢慢调回来"的容错空间**，数值必须让 7 天这个短窗口本身就是完整、
有梯度、有正反馈的体验。

> **v1.5 修订（grilling 会话结论）：存活率目标从"大概率通关"翻转为"失败是常态"**。
> 参考《赛博徒步·生死鳌太线》的核心洞察——死亡本身如果好笑、好截图，就是最强的传播素材，
> "这次能不能撑过去"的悬念比"这次能考多少分"更能驱动"再试一次"。**只抄这套高致命性哲学，
> 不抄它的四维生存值/装备子系统**，仍然只用 money/rep 两条数值轴，破产判定逻辑（§8）不变，
> 只是把目标存活率大幅下调。§9.2 的墓志铭系统就是为了让"大概率会死"这件事本身变得好玩、
> 值得截图，而不是纯粹的挫败感。

**Task 9 实测结果（600 局/策略，已固化为 `tests/sim.test.js` 断言）：**

- 「合理策略」bot（按客流备菜、正常价、缺菜用替代救单、保守买升级）：**7 天存活率 ≈ 37%**
  （落在目标 30%~45%）——多数"正常打法"的玩家也会在某天被房租/连环倒霉事件拖垮，这是刻意
  设计。survivor 终局分数分布 0~135、中位 ~41（§9 评分阈值据此设定）；
- 「躺平策略」bot（每天只备 4 份菜）：**7 天存活率 ≈ 0%**——低产出覆盖不了 $32 房租，
  缓冲（$82 起始 + $60 贷款）被烧穿，摆烂几乎必死；
- 「全斩客策略」bot（全部选 slash）：**存活率 ≈ 19%**，明显低于合理策略（斩客短期多收的现金
  被 rep 崩坏→客流萎缩→走单损耗抵消，得不偿失）；
- **关键平衡杠杆（复盘）**：① 菜品毛利（§3 上调）让好玩家满摊能盈利；② 房租 $32 + 弱贷款
  ($60) 让低产出策略必死；③ 起始 $82 偏紧，好玩家也会因前期倒霉而 ~63% 阵亡。三者联动，
  改任一处必须重跑 `npm test` 的 sim 断言确认没有连锁失衡（尤其别单纯砍 `START_MONEY`，
  优先动 `RENT_PER_DAY` 与事件惩罚）。

## 12. 美术资产规范（v1.4 修订，grilling 会话新增）

> 修订原有"全 emoji、零图片"决策：菜品、顾客、食阁场景改用**像素风插画**，
> emoji **保留作为图片加载失败时的兜底显示**（`<img onerror>` 切回对应 emoji），
> 不是二选一。原始决策的"零外部资源"精神保留在别处（仍不用字体文件、不用 CDN、
> 不用框架），这里单独为"像素画"这一类资源开一个可控的体积预算（见 03-architecture §10）。

### 12.1 视觉风格

- **风格基调**：SNES/《星露谷物语》一类的复古像素风，不是极简 8-bit，允许适度细节和光影，
  但保持"看得出是像素画"的颗粒感（不要生成成写实插画后再简单地加马赛克滤镜，
  要求生成模型直接按像素风格描述作画）。
- **调色板复用现有 CSS 变量**（`style.css` 已定义的 kopitiam 配色，保证插画和 UI 底色不冲突）：
  主色 `#1e5c46`（墨绿）、`#fdf6e3`（奶白）、`#c73e2e`（摊位红）、`#e8862f`（橙）、`#f5c542`（黄）。
  每张生成 prompt 都要显式带上这组色号，是维持几十张图风格统一的最关键手段。
- **一致性技术手段**：先生成 1 张"风格锚点图"（建议用食阁场景，因为构图最复杂、最能定调），
  之后其余素材用同一张锚点图做 image-to-image 参考（而不是每张图独立用纯文字 prompt 生成），
  大幅降低风格漂移风险。

### 12.2 素材清单（共 21 张，命名与 data.js 的 id 一一对应）

| 类别 | 数量 | 命名规则 | 构图要求 |
|---|---|---|---|
| 菜品（DISHES） | 12 | `assets/dishes/<dishId>.webp` | 单份菜品俯视/45° 特写，白瓷盘或蕉叶垫底，无背景干扰 |
| 顾客（CUSTOMER_TYPES） | 7 | `assets/customers/<typeId>.webp` | 半身像，正面或 3/4 侧脸，风格统一的"证件照"构图，方便和文字对话框并排显示 |
| 食阁场景 | 2 | `assets/scenes/cover.webp`（封面用，带摊位全景）<br>`assets/scenes/counter.webp`（营业阶段背景，摊位柜台视角） | 前者宽幅、氛围感强，适合做分享封面；后者简洁，避免和前景的顾客卡/按钮抢视觉 |

**明确不做像素画的部分**：11 种食材（`INGREDIENTS`）继续用 emoji——它们只出现在采购/备菜的
数量清单里，是操作型元素而非叙事型元素，emoji 已经够清楚，不值得为此增加 11 张素材的产出与维护成本。

### 12.3 生成与接入流程

> **search-first 复查结论**：先查过 Kenney.nl 一类 CC0 免费像素素材包，够不够直接抄现成的、
> 不用生成。结论是不采用——那些包是通用食物/角色图标（汉堡、披萨、通用 NPC），没有"咖喱鱼头"
> "阿嬷""建筑工人"这类东南亚小贩文化的具体内容，而这恰恰是本作辨识度的核心，用通用素材会
> 削弱记忆点。因此 21 张仍走定制生成，不是没查过就动手画。

1. 用 fal.ai（`fal-ai/nano-banana-pro`，高保真优先，风格锚点图不能糊）按 12.1/12.2 生成全部 21 张，
   风格锚点图（食阁场景其一）先出，其余用它做 image-to-image 参考；
2. 人工过一遍风格一致性（尤其调色板是否跑偏），不满意的重新生成而不是将就；
3. 每张图导出后转 WebP（更小体积，iOS Safari 14+/Android Chrome 均支持），
   目标单图 ≤ 15KB（21 张合计 ≤ 315KB，预算依据见 03-architecture §10）；
4. `data.js` 每个 dish/customerType 对象新增 `img` 字段（相对路径，见 12.2 命名规则）；
5. 场景图直接作为 CSS `background-image` 或 `<img>` 使用，不需要额外字段。
