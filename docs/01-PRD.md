# 《杂菜饭 Uncle》产品需求文档（PRD）

| 项 | 内容 |
|---|---|
| 产品名 | 杂菜饭 Uncle（英文副题：Cai Png Tycoon） |
| 形态 | 手机浏览器网页游戏（H5，竖屏优先），无需安装、无需登录 |
| 类型 | 文字选择驱动的轻数值经营模拟，**单局独立、7 天讲完一个结局** |
| 目标平台 | iOS Safari / Android Chrome / 桌面浏览器（次要） |
| 版本 | v1.0（MVP，本计划范围） |
| 文档状态 | 定稿，作为所有下游文档的需求基线 |

## 1. 产品愿景与定位

**叙事外壳**：邻居杂菜饭 Uncle 临时有事，把摊子托付给你看 7 天——玩家不是"经营一家永续小店"，
而是**代班一周**：每天清晨采购食材、备菜、营业接客、收档结算、升级摊位，撑过 7 天，
交出一份成绩单，故事讲完，一局结束。游戏以**文字选择**推进，配合轻量数值经营，
单天流程控制在几分钟内，**一整局（7 天）15~20 分钟即可坐着玩完**，适合手机碎片时间。

差异化卖点（游戏的「记忆点」）：

1. **Uncle 玄学报价**——杂菜饭在现实中以「没有价目表、Uncle 心情定价」闻名（社交网络长期热梗）。
   游戏把这个梗做成核心机制：每一单由玩家现场喊价（良心价/正常价/斩客价），
   不同顾客反应不同，直接影响声望与收入，且不同类型顾客"好斩"与"别斩"存在真实差异
   （详见 `02-game-design.md` §4.2 的 grilling 修订），不是无脑一档打天下。
2. **市井烟火气文案**——顾客对白带新马华语/Singlish 风味（"Uncle，加个卤蛋 leh"），
   角色鲜活，事件接地气（下雨、卫生检查、隔壁摊打价格战）。
3. **高致命性 + 死亡值得截图**——v1.5 修订（grilling 会话结论）：这局"能不能撑过 7 天"本身
   就是悬念，参考爆火产品《赛博徒步·生死鳌太线》的洞察，多数玩家会破产，但破产结局配专属
   黑色幽默墓志铭文案（见 `02-game-design.md` §9.2），失败本身好笑、好截图，驱动"再来一局"，
   而不是让人觉得"我输了很沮丧"。

## 2. 目标用户与业务目标

- **目标用户**：休闲玩家、东南亚华人文化圈用户；通过链接在微信/WhatsApp/小红书传播，
  单次点开链接就能完整玩完一局，是获客转化的关键假设。
- **业务性质**：个人作品/传播型小游戏，无商业化诉求。业务目标按优先级：
  1. 手机浏览器打开链接即玩，加载 < 2 秒（3G 网络 < 5 秒）；
  2. 单日循环好玩、有梗、可截图传播；
  3. **一整局（7 天）15~20 分钟内坐着玩完，一次点开链接就是一次完整体验**——不要求玩家
     为了"看结局"分多次回访，也不支持中途暂停恢复：关闭页面 = 这局作废，只能「重新开始」
     （详见 §4，取代此前 grilling 会话中讨论过的"周报-连续经营"方向，那个方向已废弃）；
  4. 零运维成本长期在线（静态托管）。

## 3. 核心玩法循环（一天 = 一个循环，共 7 天）

```
清晨采购 → 备菜 → 营业接客(核心) → 收档结算 → 升级商店 → 次日（第 N/7 天）
  30s        30s      2~3min           20s        20s
```

1. **清晨采购**：在预算内购买食材（米、蛋、青菜、鸡肉……），部分食材随本局声望解锁。
2. **备菜**：将食材烹成菜品（每菜消耗固定食材），受备菜容量限制，决定「炒什么、炒几份」。
3. **营业接客**：顾客逐个到来（文字互动）。每位顾客有类型、性格、订单：
   - 有菜 → 出餐 → **玄学报价**三选一 → 顾客按类型/概率反应（付钱/砍价/掉头走/发帖）；
   - 缺菜 → 选择「推荐替代」或「道歉送客」。
4. **收档结算**：收入、食材损耗（剩菜隔夜作废，除非有冰箱）、房租、贷款利息、声望变动汇总；随机夜间事件。
5. **升级商店**：用利润购买摊位升级（冰箱/大锅/招牌/咖啡机/帮手……），升级只在本局内有效。

## 4. 单局目标与结局（7 天讲完，没有"下一局"的延续）

- **本局进度**：money / rep / upgrades 全部从初始值开始，只在这 7 天内累积，
  **不会带到下一局**——每局都是一次独立的、完整的代班故事，重开 = 一切归零重新讲一遍。
- **声望等级**（0→120+，路边小摊 → 巷口熟客 → 街坊招牌 → 排队名摊 → 本地传奇 → 全城最强杂菜饭）：
  本局内解锁新食材/新顾客类型/新事件的节奏控制器。前几档（15/30/55）在合理打法下 7 天内应当可及；
  85/120 是留给"打得非常好"的玩家的挑战性高分，而不是必须达成的内容解锁点——具体阈值待
  `sim.js`（Task 9）跑出 7 天口径的数据后校准，见 `02-game-design.md` §11。
- **结局（第 7 天收档后，撑过去了）**：一次性生成成绩单——评分（S/A/B/C）+ Uncle 人设标签（见
  `02-game-design.md` §9/§9.1），rep 若摸到 120 会在成绩单里给一个"全城最强"的特殊高分评价，
  **不是弹窗打断、不是"之后继续玩"，就是这局故事的结尾**。之后只有一个选择：「再来一局」（从头开始）。
- **失败线（大概率会走到这一步）**：现金破产 → 一次「阿财叔贷款」续命机会 → 再次资不抵债 =
  本局提前结束，展示专属墓志铭文案（`02-game-design.md` §9.2），同样只能「重新开始」——
  **v1.5 修订**：目标存活率大幅下调（见 §7），破产不再是边缘情况而是这局设计的常态结果之一。

## 6. 非功能需求

- **性能**：html+css+js < 200KB（不含字体，用系统字体）+ 像素画资产 ≤ 315KB（21 张 WebP，
  单图 ≤15KB，推导见 03-architecture §10），总计仍满足 3G < 5s 的加载目标；无外部 CDN 依赖
  （可被墙/CSP 环境照常运行，图片资产随仓库一起托管，不依赖第三方图床）。
- **兼容**：iOS Safari 15+、Android Chrome 90+；`100dvh` 需有回退。
- **可维护**：运行时零 npm 依赖；核心逻辑纯函数、100% 可单测；数值全部集中在一张数据表文件。
- **持久化**：**不做中途游戏状态持久化**（刷新/关闭 = 本局作废）；F12 的历史最佳成绩若采纳，
  只是一个独立的、极简的小对象，不涉及版本迁移这类复杂度。
- **可访问性**：按钮点击区 ≥ 44px；文字对比度达标。

## 7. 成功度量（上线后观察）

- 手机真机打开到可玩 < 2s；
- **一整局（7 天）单次坐着通关时长 15~20 分钟**；
- 千局自动模拟（7 天口径，**v1.5 修订：目标从"大概率通关"翻转为"失败是常态"**，见
  `02-game-design.md` §11）：合理策略存活率目标 30%~40%（撑过去的局里 A 档以上比例 ≥ 60%）；
  躺平策略存活率 ≤ 10%，和合理策略拉开明显差距（数值有区分度，见 04-test-plan）。

## 8. 风险与对策

| 风险 | 对策 |
|---|---|
| 数值崩坏（太easy躺赢/太难劝退，尤其压缩到 7 天后节奏完全变了） | 建 `sim.js` 千局模拟器（7 天口径），数值改动跑回归（04-test-plan §4） |
| 文案趣味性不足 | 文案集中在 `data.js`，与逻辑解耦，可独立迭代打磨 |
| 移动端适配翻车 | Playwright iPhone 视口 E2E 走全流程 + 截图比对 |
| "不能暂停"被玩家误当作 bug（比如手机来电打断后回来发现要重开） | 结局页/封面文案明确提示"本局不支持中途保存，请一次性玩完"；额外加 `beforeunload` 确认拦截（F18），减少误触退出 |
| 21 张 AI 生成像素画风格不统一（调色板/构图跑偏） | 先出 1 张风格锚点图，其余用 image-to-image 参考它生成，而非各自独立文字生成（见 02-game-design §12.1） |
| 图片资产拖慢加载、超出体积预算 | 单图 ≤15KB 硬指标 + WebP 压缩，超标必须重新压缩，见 03-architecture §10 的预算推导 |
| 高致命性（30%~40% 存活率）劝退轻度/首次玩家，而不是激发"再来一局" | 墓志铭文案（F17）把破产变成好笑、值得截图的内容而非单纯挫败感；day 1 客流预热 + 每日手感反馈（02-game-design §9.3）降低"完全摸黑猝死"的不公平感；上线后若实测劝退率过高，先调 §11 的调参方向（加重事件而非砍 START_MONEY），而不是马上放弃高致命性设计 |

## 9. 明确不做（YAGNI）

服务端/账号系统、排行榜、多语言、内购广告、Canvas/WebGL 渲染、框架（React 等）、打包器、
**跨局存档/中途暂停恢复、赛季制、无尽模式**。
